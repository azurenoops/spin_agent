# ADR-008: Strategic Roadmap — v1.3.0 through v1.5.0

**Status:** proposed

**Date:** 2025-07-19

## Context

Teamwork v1.2.0 is a mature agent-native development template with 17 agents, 10 skills, a Go CLI with 20+ commands, 13 MCP integrations, and file-based coordination. Eight backlog issues remain open (#39, #40, #41, #44, #46, #55, #61, #62).

The AI agent framework space is crowded and moving fast. To guide the next three minor releases, we need a clear-eyed comparison with competing/adjacent frameworks, an honest accounting of our gaps, and a strategy that doubles down on Teamwork's unique strengths rather than chasing features that don't fit our architecture.

This ADR captures the competitive analysis, gap assessment, strategic themes, and feature recommendations for v1.3.0, v1.4.0, and v1.5.0.

### 1.2 Key Competitive Observations

**CrewAI** is the closest competitor in philosophy (roles, delegation, workflows) but operates as a Python runtime SDK. Its training system — where crews learn from human feedback and improve over iterations — is the most notable capability Teamwork lacks. CrewAI's memory uses embeddings for semantic search, which enables agents to recall relevant past context without manual tagging.

**LangGraph** is the gold standard for workflow resilience. Its checkpointing system persists full state at every graph node, enabling resume after failure, time-travel debugging, and replay. Its graph-based model (cycles, conditionals, parallel branches) is fundamentally more expressive than Teamwork's linear step sequences. However, LangGraph is a low-level SDK — it provides building blocks, not roles or conventions.

**AutoGen** pioneered multi-agent conversation patterns but is moving toward a more modular architecture (AutoGen 0.4+). Its "group chat" model — where agents discuss and negotiate — is philosophically different from Teamwork's handoff model. The conversation approach is powerful for ambiguous problems but produces less auditable output than artifact-based handoffs.

**Agno** has the most mature memory system with database-backed persistent storage, session memory, and structured retrieval. Its "agent teams" concept with leader/member hierarchy is similar to Teamwork's orchestrator model but with runtime delegation.

**SWE-agent, OpenHands, Devon** are autonomous agents, not frameworks. They are potential *users* of Teamwork-style structure, not competitors. Their approach (give an AI full access and let it work autonomously) is the opposite of Teamwork's philosophy (structured roles, quality gates, human approval).

**Cursor Rules and Aider** are the closest to Teamwork's "configuration over code" philosophy. Cursor Rules (`.cursor/rules/`) are file-based behavioral instructions — essentially a simplified version of what Teamwork agents do. Aider's repository map (tree-sitter based context selection) is an interesting approach to the context window problem that Teamwork could learn from.

## 2. Gap Analysis

### 2.1 Critical Gaps (competitors universally have, high user impact)

| Gap | Backlog Issue | Who Does It Well | Impact If Not Addressed |
|---|---|---|---|
| **Workflow checkpointing and resume** | #39 | LangGraph (built-in), CrewAI (crew replay) | Long-running workflows are fragile; any interruption means starting over. As workflows get more complex, this becomes a dealbreaker. |
| **Structured feedback/learning loop** | #44 | CrewAI (training system) | Agents repeat the same mistakes across sessions. Memory captures *facts* but not *behavioral corrections*. The reviewer identifies issues that the coder never learns from. |
| **Context distillation** | #62 | Aider (repo map), CrewAI (summary memory) | As workflows accumulate handoffs, context windows fill up. Agents waste tokens re-reading verbose handoffs when a compressed summary would suffice. |

### 2.2 Important Gaps (most competitors have, moderate user impact)

| Gap | Backlog Issue | Who Does It Well | Impact If Not Addressed |
|---|---|---|---|
| **Semantic search over memory** | #41 | CrewAI (embeddings), Agno (DB queries) | Memory is only useful if you can find the right entry. Tag-based lookup scales to ~100 entries; beyond that, agents miss relevant context. |
| **Example projects** | #55 | Everyone except Teamwork | New users have to imagine how Teamwork works in practice. No worked examples = high friction adoption. Every competing framework has "cookbook" or "examples" repos. |
| **Workflow analytics/reporting** | #61 | LangGraph (Studio traces), CrewAI (execution logs) | Metrics are collected but not surfaced as actionable insights. Teams can't answer "are our workflows getting faster?" or "which role is the bottleneck?" |
| **Programmatic state query** | #40 | LangGraph (state inspection API) | CLI-only access to state means external tools (dashboards, CI, bots) can't query workflow status programmatically. |

### 2.3 Defensible Absences (competitors have but Teamwork should NOT add — see §5)

| Capability | Why NOT to Add |
|---|---|
| Agent-to-agent runtime delegation | Copilot CLI is the runtime; Teamwork defines structure, not execution |
| Embedding/vector database | Adds infrastructure dependency; file-based is a feature |
| Own LLM execution runtime | Teamwork is runtime-agnostic by design |
| Autonomous mode | Human-as-executive is a core differentiator |
| Conversation-based agent interaction | Handoff-based coordination is more auditable and debuggable |

## 3. Differentiation Analysis — What Teamwork Has That Competitors Don't

### 3.1 Unique Strengths to Protect and Amplify

**1. Template-first, runtime-agnostic architecture**
No other framework operates as a *template*. CrewAI, LangGraph, AutoGen, and Agno are all Python SDKs — you write code to define agents and workflows. Teamwork defines everything in Markdown and YAML files that any AI tool can read. This means Teamwork works today with Copilot CLI, Claude Code, Cursor, Windsurf, Aider, or any future tool. This is an extraordinary moat: as AI runtimes consolidate or fragment, Teamwork users are not locked in.

**Amplify by:** Making it trivially easy to adopt (examples, onboarding), and ensuring the template files themselves are the canonical source of truth — not the CLI.

**2. Enforced role separation with quality gates**
No competing framework enforces that the agent writing code is different from the agent reviewing it. CrewAI allows role definition but doesn't prohibit self-review. Teamwork's boundary system (✅ Always / ⚠️ Ask first / 🚫 Never) is unique in the space. Quality gates between steps are another unique concept — a structural guarantee that output is validated before the next phase begins.

**Amplify by:** Making quality gates configurable and observable. Add more gate types. Surface gate pass/fail rates in analytics.

**3. File-based, git-trackable coordination**
Every state file, handoff, and memory entry is a file in the repo. This means workflow state is versioned, diffable, code-reviewable, and backed up automatically. LangGraph stores state in databases or memory. CrewAI uses in-process memory. Teamwork's approach is uniquely transparent.

**Amplify by:** Adding richer querying on top of files (without abandoning files for a database).

**4. GitHub-native structure**
`.github/agents/` and `.github/skills/` are first-class GitHub Copilot concepts. Teamwork agents appear in Copilot's agent dropdown. Skills are invocable via `/skill-name`. No other multi-agent framework integrates this deeply with GitHub's native AI infrastructure.

**Amplify by:** Tightening GitHub integration (issue-driven workflows, PR-linked state, milestone-aware orchestration).

**5. Behavioral contracts, not code**
Agent files are *specifications*, not implementations. They define what an agent should do, what it must not do, what quality bar it must meet, and when it should escalate. This is closer to a team charter than a software module. The abstraction level is fundamentally different from CrewAI (Python class), LangGraph (Python function), or AutoGen (agent config object).

**Amplify by:** Making contracts more precise and testable (validate that agent output meets the quality bar defined in its contract).

### 3.2 Unique Value Proposition (One Sentence)

> Teamwork is the only framework that structures AI agent collaboration through behavioral contracts and file-based coordination, working with any AI runtime and enforcing role separation, quality gates, and human authority without runtime lock-in.

## 4. Strategic Themes for v1.3.0 – v1.5.0

### Theme 1: Workflow Resilience — "Workflows that survive the real world"

**Why it matters:** Today, if an agent session crashes mid-workflow, the workflow is in a broken state. If a step fails, the only option is human intervention or restarting. As teams adopt Teamwork for production work, workflows must be robust against interruptions, failures, and long-running multi-day tasks.

**What competitors do:** LangGraph has the best story here — full state checkpointing at every node, automatic resume, time-travel debugging. CrewAI has crew replay for restarting from specific points. Both treat resilience as foundational, not optional.

**What Teamwork should do differently:** Implement checkpointing as file-based snapshots (staying true to the git-trackable model), not database-backed. Add step-level retry with configurable policies. Make state queryable via CLI flags and JSON output for CI/tooling integration. Keep it template-first: resilience configuration lives in `config.yaml`, not in code.

### Theme 2: Adaptive Memory — "Agents that learn and remember the right things"

**Why it matters:** Teamwork has memory (patterns, antipatterns, decisions, feedback in YAML files), but it's passive — agents must know to look for entries and manually search by domain tag. As projects grow, memory becomes a haystack. More critically, Teamwork lacks a learning loop: when a reviewer finds issues, those corrections don't systematically flow back to influence future coder behavior.

**What competitors do:** CrewAI has a training system where crews learn from scored human feedback. Agno has database-backed memory with structured retrieval. Aider uses tree-sitter to build a repository map that adapts as code changes.

**What Teamwork should do differently:** Stay file-based but add intelligence at the *access* layer. Context distillation compresses verbose handoffs into lean summaries. Feedback loops capture reviewer corrections as typed memory entries that coders read at session start. Semantic search should be an MCP server (not an embedded database), keeping the "template + tools" architecture clean. Memory relevance scoring surfaces the 3-5 most pertinent entries for a given task rather than dumping everything.

### Theme 3: Adoption Velocity — "Zero to productive in one session"

**Why it matters:** Every competing framework has examples, cookbooks, quickstarts, and playgrounds. Teamwork has none (#55). The template model is powerful but abstract — a new user reading agent files and skill definitions needs to *see it work* to understand the value. Without worked examples, Teamwork remains appealing in theory but opaque in practice.

**What competitors do:** CrewAI has 50+ example crews. LangGraph has a cookbook with ~30 recipes. AutoGen has 20+ notebooks. Even Cursor Rules has a community-maintained directory. These aren't nice-to-haves; they're how developers evaluate and adopt frameworks.

**What Teamwork should do differently:** Ship 3-4 complete example projects spanning different tech stacks (a CLI tool, a web API, a full-stack app). Each example should include a completed workflow trace (state files, handoffs, memory entries) so users can see exactly what a workflow looks like end-to-end. Add a `teamwork demo` command that walks through a mini-workflow interactively. Invest in custom workflow definitions so users can extend beyond the 10 built-in skills.

### Theme 4: Observable Quality — "Trust through transparency"

**Why it matters:** The core promise of Teamwork is that agent output is trustworthy because of role separation and quality gates. But today, that trust is asserted, not demonstrated. Teams can't see aggregate quality metrics, gate pass/fail rates, or patterns in where workflows get stuck. The feedback loop (#44) is missing, so quality doesn't improve over time.

**What competitors do:** LangGraph Studio provides visual traces and step-level debugging. CrewAI's training system explicitly improves quality over iterations. Most frameworks have observability integrations (LangSmith, Langfuse).

**What Teamwork should do differently:** Make quality visible in the CLI. `teamwork metrics` should answer "what percentage of reviewer gates pass on first try?" and "which steps take the longest?" Ship a structured feedback loop where reviewer comments become typed memory entries. Add secrets scanning as a quality gate (#46). Publish metrics as Markdown reports that can be committed or posted to PRs — staying true to the file-based, transparent philosophy.

## 5. Feature Recommendations by Release

### v1.3.0 — "Resilient Workflows"

*Theme: Make workflows survive interruptions, support retries, and expose state programmatically.*

| # | Feature | Source | Size | Why This Release |
|---|---------|--------|------|------------------|
| 1 | **Workflow checkpointing and resume** | #39 | L | Most critical gap. Every session crash today loses workflow progress. Checkpoints are file-based snapshots saved before each step transition, enabling `teamwork resume <workflow-id>`. |
| 2 | **Programmatic state query and filtering** | #40 | M | Unblocks CI integration and external tooling. `teamwork status --json --filter status=active,role=coder`. Pairs naturally with checkpointing. |
| 3 | **Step retry with backoff** | NEW | M | When a step fails (tests don't pass, lint fails), allow `teamwork retry <workflow-id>` to re-enter the current step rather than failing the whole workflow. Configurable max retries in `config.yaml`. |
| 4 | **Automated secrets scanning gate** | #46 | S | Low-effort, high-trust. Add a quality gate that runs pattern-based secrets detection on handoff artifacts and committed files before allowing step advancement. Uses Semgrep MCP when available, falls back to regex patterns. |
| 5 | **Workflow analytics and aggregate reporting** | #61 | M | Surface the metrics already being collected. `teamwork metrics report` generates a Markdown summary: avg duration per role, bottleneck identification, gate pass rates, workflow completion rates. |
| 6 | **Conditional step skipping (enhanced)** | NEW | S | Extend `skip_steps` in config to support condition-based skipping (e.g., skip security-auditor if diff is docs-only). The engine evaluates conditions against the workflow context rather than requiring static per-type configuration. |

**Total: 6 features, ~2L + 3M + 1S ≈ 7-8 issues**

### v1.4.0 — "Adaptive Memory"

*Theme: Make memory intelligent — feedback loops, context compression, and relevant retrieval.*

| # | Feature | Source | Size | Why This Release |
|---|---------|--------|------|------------------|
| 1 | **Structured feedback loop (Reviewer → Coder)** | #44 | L | The highest-ROI learning feature. When a reviewer leaves corrections, those corrections are captured as typed feedback entries in `.teamwork/memory/feedback.yaml` with source PR, domain tags, and severity. Coder agent files are updated to read feedback entries for their current domain at session start. |
| 2 | **Automatic context distillation** | #62 | L | Handoffs grow verbose over multi-step workflows. Add `teamwork distill <workflow-id>` that produces a compressed context summary from all handoffs in a workflow — key decisions, open questions, artifacts produced — sized to fit a single prompt. The orchestrator auto-generates distilled context after step 3+ of any workflow. |
| 3 | **Semantic search over memory (MCP server)** | #41 | M | New MCP server (`teamwork-mcp-memory`) that indexes `.teamwork/memory/*.yaml` entries and supports semantic similarity search. Uses local embeddings (sentence-transformers) — no external API dependency. Agents call `memory_search(query)` instead of manually scanning YAML files. |
| 4 | **Cross-workflow pattern extraction** | NEW | M | After a workflow completes, the orchestrator analyzes the workflow's handoffs and metrics to extract reusable patterns: "what worked," "what took too long," "what was blocked and why." These are auto-proposed as memory entries (human approves before they're committed). |
| 5 | **Memory relevance scoring** | NEW | S | When an agent starts a step, automatically surface the 3-5 most relevant memory entries based on the workflow type, current role, domain tags, and handoff context. Output as a "Relevant Context" section in the `teamwork next` output. |
| 6 | **Handoff compression format** | NEW | S | Define a "lean handoff" format alongside the full handoff: a structured YAML summary (≤500 tokens) that captures decisions, artifacts, and open questions without prose. Steps 4+ in long workflows read the lean handoff by default, with the full version available on request. |

**Total: 6 features, ~2L + 2M + 2S ≈ 7-8 issues**

### v1.5.0 — "Adoption & Ecosystem"

*Theme: Make Teamwork easy to adopt, extend, and share.*

| # | Feature | Source | Size | Why This Release |
|---|---------|--------|------|------------------|
| 1 | **Example projects** | #55 | L | Three complete example projects: (a) Go CLI tool, (b) Node.js/Express API, (c) Python FastAPI app. Each includes a `.teamwork/` directory with a completed workflow trace — state files, handoffs, memory entries — so users can read a real end-to-end execution. |
| 2 | **Custom workflow definitions** | NEW | L | Allow users to define new workflow types in `.github/skills/*/SKILL.md` and have the engine recognize them automatically. Today, workflow types are hardcoded in `engine.go`. | 3 | **`teamwork import` command** | NEW | M | Import agents, skills, or memory from another Teamwork-enabled repository (or a curated registry). `teamwork import --from github.com/org/repo --agents api-agent --skills api-workflow`. Enables community sharing and team-to-team knowledge transfer. |

**Total: 6 features, ~2L + 3M + 1S ≈ 7-8 issues**

## 6. What NOT to Build

These capabilities exist in competing frameworks but should NOT be added to Teamwork. For each, the rationale explains why.

### 6.1 Own LLM Runtime / Agent Execution Engine

**Who has it:** CrewAI, LangGraph, AutoGen, Agno, SWE-agent, OpenHands

**Why not:** Teamwork's entire value proposition is runtime-agnostic behavioral contracts. Building an execution engine would duplicate what Copilot CLI, Claude Code, and Cursor already do, lock users into a runtime, and transform Teamwork from a lightweight template into a heavyweight SDK competing with well-funded incumbents. The template model is a moat — don't fill it in.

### 6.2 Embedded Vector Database / Embedding Store

**Who has it:** CrewAI (long-term memory), Agno (database-backed memory)

**Why not:** Adding a database dependency violates the "files in your repo" principle. It creates setup friction (install a DB, configure embeddings, manage indices), breaks the git-trackable property, and adds infrastructure that must be maintained. Instead, semantic search should be an *optional* MCP server (`teamwork-mcp-memory`) that indexes the existing YAML files. Users who want it install the server; users who don't keep the simple file-based approach.

### 6.3 Agent-to-Agent Runtime Communication / Delegation Protocol

**Who has it:** CrewAI (delegation), AutoGen (group chat), Agno (team leader dispatch)

**Why not:** Teamwork agents communicate through file-based handoffs, not runtime messages. The orchestrator (or copilot-instructions.md auto-dispatch) determines which agent runs next. Building a delegation protocol would require a running process to route messages — which is what an execution runtime is. Let Copilot CLI's `task` tool handle agent dispatch; Teamwork defines the roles and sequence.

### 6.4 Autonomous / Unsupervised Mode

**Who has it:** SWE-agent, OpenHands, Devon

**Why not:** "Human as executive" is not a limitation — it's the core design decision (ADR-001). Teamwork's quality guarantee comes from human approval gates. Autonomous mode removes the mechanism that makes agent output trustworthy. Teams that want full autonomy should use SWE-agent or OpenHands directly; Teamwork serves teams that want structured collaboration with human oversight.

### 6.5 IDE-Specific Plugins or Extensions

**Who has it:** Cursor (built-in), Cline (VS Code extension), Aider (terminal-native)

**Why not:** Teamwork already works natively in VS Code (via Copilot agents/skills), Cursor (via agent files), and Claude Code (via CLAUDE.md + file reading). Building IDE-specific plugins fragments the maintenance surface and ties features to specific editors. The `.github/` file structure is the universal integration point. If an IDE can read markdown files, it can use Teamwork.

### 6.6 Conversation-Based Agent Interaction

**Who has it:** AutoGen (group chat), CrewAI (agent discussion)

**Why not:** Handoff-based coordination produces auditable, deterministic artifacts. Conversation-based interaction produces chat logs that are hard to review, diff, or replay. Teamwork's handoff model is more structured and transparent. The tradeoff is reduced flexibility for ambiguous problems — but that's what the spike workflow and human escalation are for.

### 6.7 SWE-bench Optimization / Benchmark Chasing

**Who has it:** SWE-agent (built for SWE-bench), OpenHands (benchmarked)

**Why not:** Teamwork is a process framework, not a coding agent. Benchmarks measure single-agent task completion; Teamwork's value is in multi-agent coordination, quality gates, and human oversight. Optimizing for SWE-bench would warp priorities toward autonomous single-agent performance, which is the opposite of Teamwork's architecture.

## 7. Release Sequencing Rationale

The release order is intentional:

**v1.3.0 (Resilient Workflows) comes first** because without checkpointing and retry, workflows are fragile. This is table-stakes infrastructure that everything else depends on. A learning system (v1.4.0) is less valuable if workflows crash before completing. Examples (v1.5.0) are less compelling if the demonstrated workflows break easily.

**v1.4.0 (Adaptive Memory) comes second** because once workflows reliably complete, the data they produce (handoffs, metrics, feedback) becomes the foundation for learning. Cross-workflow pattern extraction requires completed workflows to extract from. The feedback loop requires multiple completed review cycles to produce value.

**v1.5.0 (Adoption & Ecosystem) comes third** because the features it delivers (examples, custom workflows, import, composition) benefit from the stability of v1.3.0 and the intelligence of v1.4.0. Example projects should demonstrate checkpointing and memory features. Custom workflows should leverage the resilient engine. Import/sharing is more valuable when there's a richer ecosystem to share.

The "Observable Quality" theme is woven across all three releases rather than concentrated in one: secrets scanning in v1.3.0, feedback loops in v1.4.0, reporting in v1.5.0.

## 9. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Checkpointing implementation is complex | Medium | High | Start with simple "snapshot before each step transition" — don't attempt LangGraph-level time-travel in v1.3.0 |
| Semantic search MCP server adds Python dependency | Low | Medium | It's optional (MCP server, not embedded). Users who don't install it lose semantic search, not basic memory |
| Custom workflow definitions are hard to validate | Medium | Medium | Start with strict frontmatter schema; require all steps to reference known roles; validate with `teamwork validate` |
| Example projects become stale | Medium | Medium | CI test that validates example `.teamwork/` directories; treat examples as first-class tested artifacts |
| Scope creep — individual features grow beyond minor-release size | High | Medium | Enforce issue decomposition via planner; each feature = 1-3 implementation issues; cut scope before cutting releases |

## Alternatives Considered

| Alternative | Why It Was Rejected |
|---|---|
| **Build a Python SDK alongside the template** | Fragments the project across two ecosystems. Runtime-agnostic is the moat; a Python SDK undermines it. |
| **Add graph-based workflows (like LangGraph)** | Complexity explosion for marginal benefit. Most development workflows are inherently sequential with occasional retries. Workflow composition (v1.5.0) provides branching without graph complexity. |
| **Partner with a specific AI runtime** | Lock-in destroys the template model's value. Better to stay agnostic and let copilot-instructions.md handle runtime-specific integration. |
| **Build a web dashboard instead of CLI reporting** | Adds hosting/deployment complexity. Markdown reports (file-based, git-trackable) align with Teamwork's philosophy. A web dashboard could be a community contribution later. |
| **Prioritize adoption (v1.5.0 themes) before resilience (v1.3.0)** | Examples demonstrating fragile workflows would hurt adoption more than help it. Resilience first ensures examples showcase a robust system. |

## Decision

Adopt the three-release roadmap as described: v1.3.0 (Resilient Workflows), v1.4.0 (Adaptive Memory), v1.5.0 (Adoption & Ecosystem). Create GitHub milestones and decompose each release's features into implementation issues for the Planner to process.
